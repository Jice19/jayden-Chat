```
//面试写法
function timeout(fn,timeout){
	return new Promise((resolve,reject) => {
			let timer = null
			timer = setTimeout(() => {
					fn()
			},timeout)
			if(timer){
				resolve(1)
			}else{
				reject(2)
			}
	})
}

//正解：
function timeout(fn, timeout) {
  return new Promise((resolve, reject) => {
    // 1. 设置超时定时器：超时直接 reject
    const timer = setTimeout(() => {
      reject(new Error('请求超时'));
    }, timeout);

    // 2. 执行 fn（返回 Promise），成功 resolve，失败 reject
    fn()
      .then(res => {
        clearTimeout(timer); // 成功后清除定时器，避免超时逻辑执行
        resolve(res);
      })
      .catch(err => {
        clearTimeout(timer); // 失败后也清除定时器
        reject(err);
      });
  });
}
```

### 1、langgraph节点如何设计的？解决了什么问题？

>
>
>我当时想实现一个「统一输入、多模态输出」的对话入口，用户只需要发一句话，系统就能自动判断是要文字回答还是生成图片，不用切换界面。为了让流程更可控、可扩展，我用 LangGraph 做了一个状态驱动的工作流来实现这个功能。
>
>整体设计是这样的：
>
>1. **状态管理**：用 LangGraph 的 `Annotation` 定义了全局共享状态 `UnifiedState`，把用户消息、意图类型、对话历史、文本回复、图片 URL 这些字段统一管理，所有节点都通过状态通信，避免了全局变量和状态混乱。
>
>2. **节点拆分**：我把逻辑拆成了三个职责单一的节点：
>
>   - `router` 节点：用千问 Plus 做意图识别，通过固定的 Prompt 模板，把用户输入解析成 `text`/`image`/`both` 三种意图，同时提取生图用的英文提示词。
>   - `text` 节点：调用千问 Plus 生成文字回复，支持上下文历史对话。
>   - `image` 节点：调用阿里云 `qwen-image-2.0` 多模态模型生成图片。
>
>   
>
>3. **流程编排**：用 `addConditionalEdges` 实现动态路由，根据意图自动分发流程：纯文本请求直接走 text 节点，纯图片请求直接走 image 节点，图文混合请求会先执行 text 节点生成文字回复，再自动触发 image 节点生成图片。
>
>这个设计主要解决了两个问题：
>
>- 用声明式的图结构替代了硬编码的 if-else，让意图路由的逻辑更清晰，后续加新的意图类型（比如语音、视频）也很方便扩展。
>- 节点职责解耦，文本生成和图片生成的逻辑完全分开，后续可以单独优化某一个节点的能力，不用改动其他部分。

### 2、意图识别的模型是如何选择的？

>这两个部分我用的是阿里云的不同模型，选型的时候主要考虑了我的场景需求和成本：
>
>- **意图识别**：用的是 `qwen-plus` 文本大模型。因为意图识别本质上是文本分类 + 结构化输出的任务，用通用文本模型配合固定的 Prompt 模板，就能稳定解析出意图类型和生图提示词，而且千问 Plus 的响应速度和调用成本都很合适，对前端场景来说足够用了。
>- **图片生成**：用的是阿里云的 `qwen-image-2.0` 多模态生成模型。当时对比了几个生图模型，它的优势是：和阿里云的 API 体系适配，不用额外对接新的服务；而且中文理解和图片生成效果在测试里表现最好，还支持设置图片尺寸、水印这些参数，刚好满足我的需求。
>
>其实一开始我也考虑过用更复杂的多模态模型直接处理，但考虑到成本和复杂度，最后还是拆成了「文本模型做意图判断 + 专用生图模型做生成」的方式，这样既降低了整体的调用成本，也让每个模块的逻辑更清晰

### 3、实例化开销是如何优化的？数据指标如何采集？

>优化前，我每次处理用户请求，都会**重新创建 LLM 实例、重新编译 LangGraph**，导致大量重复创建、重复连接，性能开销非常大。
>
>后来我用了**闭包 + 单例缓存**的思路来解决：
>
>1. 我把 LLM 实例的创建，放到了 `buildGraph` 这个外部函数里，让它**只初始化一次**。
>2. 然后通过**闭包**，让 router、text、image 这三个内部节点**共享同一个 LLM 实例**，不会重复创建。
>3. 同时我把编译好的图也做了单例缓存，整个应用只编译一次，之后全部复用。
>
>这样一来，所有节点都复用同一个 LLM 连接，**避免了重复实例化带来的性能损耗**。
>
>实际测试下来，重复实例化的开销减少了大概 **60%**，请求速度明显变快，资源占用也更低
>
>- 指标采集：
>
>- > 我用 **console.time 统计耗时**，对比优化前后连续请求的平均时间。
>  >
>  > 优化前每次都要新建 LLM、编译图，耗时很高；
>  >
>  > 优化后只实例化一次，**整体耗时减少约 60%**。

### 4、虚拟列表不定高以及优化

>我这个聊天场景的虚拟列表，核心痛点是 **消息高度不固定**，特别是图片加载、SSE 流式输出的时候，消息高度会动态变化，非常容易导致 **滚动错位、内容跳屏**。
>
>针对这个问题，我做了三层核心优化，全部是自己封装的逻辑：
>
>#### - 不定高缓存机制（解决布局塌陷）
>
>我给每条消息维护了一个 **高度缓存表**，记录它的 **默认占位高度**（比如 50px）、**真实高度**、以及相对于父容器的 **top/bottom 位置信息**。
>
>- 初始化时先用默认高度占位，防止列表布局塌陷；
>- 等消息 DOM 渲染完成后，通过 `getBoundingClientRect` 获取真实高度，更新到缓存里。
>
>#### - 滚动锚定（解决高度变化导致的跳屏）
>
>当图片加载完成，消息高度从 50px 变成 100px 时，会直接影响下方所有消息的位置。
>
>为了不让用户正在看的消息突然跳走，我用 **IntersectionObserver** 监听消息可见性，结合缓存里的高度差值（dValue），动态计算并调整列表的 **scrollTop**。
>
>- **原理**：`scrollTop += 高度差值`
>- 这样用户视口会跟着消息一起下移，保证当前看的内容始终在视野里。
>
>#### -  防抖防闪烁（解决SSE 频繁追加）
>
>Mac 端 SSE 是流式输出，消息会一条接一条进来，频繁的高度变化会导致列表疯狂跳动。
>
>所以我给高度更新和 scroll 调整加了 **防抖延迟**（比如 100ms），只有当高度变化稳定一段时间后，才真正去更新布局和滚动位置。
>
>#### 追问：为什么要用 IntersectionObserver 而不是 scroll 事件？
>
>> 因为 scroll 事件是高频触发的，直接操作 DOM 会导致重排重绘，非常卡。
>>
>> IntersectionObserver 是异步、非阻塞的，只在消息进入 / 离开视口时触发，性能开销更小，而且能精准处理动态高度带来的布局偏移。
>
>#### 追问：dValue 具体是怎么计算的？有什么用？
>
>> **dValue = 真实高度 - 预设高度**。
>>
>> 它的核心作用是计算**高度变化量**。比如图片从 50px 变成 100px，dValue 就是 50。
>>
>> 滚动锚定时，`scrollTop += dValue` 就能精准推走被挤压的内容，保证用户体验不中断。

### 5、SSE断线重连如何保证的内容连续？服务端如何实现的重连，缓存如何处理

>- 前端侧
>
>我当时的核心痛点是：用户在 SSE 流式输出过程中断网、刷新，会导致消息中断，再连上时需要重新请求大模型，体验很差。
>
>我用了「**消息 ID + 本地缓存 + 续传**」的方案来解决：
>
>1. **消息带唯一 ID**：后端返回的每一条增量消息，都带一个递增的 messageId。
>2. **前端缓存已接收消息**：我把当前会话的 messageId 存在 localStorage 里，实时更新。
>3. **断线自动重连**：我用了微软基于 fetch 封装的流处理库，它会监听 onerror 事件，断网后自动重试；重连时，会把最后收到的 messageId 作为参数发给后端。
>4. **后端从断点续传**：后端收到 messageId 后，会从该 ID 之后的消息继续推流，用户完全不用重新生成，体验无感知。
>
>另外，我当时没直接用原生的 EventSource，而是用 fetch + ReadableStream 来实现，因为它的错误处理和自定义 headers 更灵活，更适合做这种带参数的续传逻辑。
>
>- 服务端侧
>
>> 我理解的后端缓存，核心目标是**支持断点续传，避免用户重连后重新请求大模型**。
>>
>> 我的设计思路是这样的：
>>
>> 1. **缓存分层设计**：
>>
>>    - **热数据（正在推流的会话）**：存在 **Redis** 里，用 sessionId 作为 key，存一个有序集合，按 messageId 存已经生成的增量内容。这样重连时可以快速定位到断点，毫秒级响应。
>>    - **冷数据（已结束的会话）**：异步持久化到数据库，防止 Redis 内存溢出。
>>
>>    
>>
>> 2. **断线续传逻辑**：
>>
>>    - 用户断网时，后端感知到连接断开，不会立刻清空缓存，而是设置一个过期时间（比如 5 分钟）。
>>    - 重连时，用户带上最后收到的 messageId，后端从 Redis 里取出该会话的缓存，从该 ID 之后继续推流。
>>    - 如果缓存过期了，或者用户长时间没回来，再从大模型重新生成，或者提示用户会话已过期。
>>
>>    
>>
>> 3. **为什么不用纯内存？**
>>
>>    因为服务端实例可能会重启，纯内存会丢数据；Redis 既支持高并发读写，又能设置过期时间，刚好适合这种临时缓存的场景。

### 6、监控SDK

>面试官您好，当时项目里已经有用户行为埋点体系，但缺少前端性能指标和异常错误的自动化采集，开源方案比如 Sentry 虽然成熟完善，但和我们现有埋点体系数据不通，无法形成用户行为 — 性能 — 异常的全链路分析，同时为了深入理解浏览器监控底层原理，我决定自研一套轻量的前端性能与异常监控 SDK。
>
>在设计上我主要做了这几方面亮点：
>
>第一，性能指标采集。基于浏览器原生 PerformanceObserver API 实现事件驱动采集，获取 FCP、LCP、资源加载等核心指标，同时做了智能数据清洗，过滤重复、无效、高频冗余数据，避免无效上报带来性能损耗。
>
>第二，上报机制做了高可靠设计。我采用了缓冲队列机制，数据攒够 20 条后批量上报；同时把数据暂存到 localStorage 做持久化，考虑到它 5MB 左右的容量限制，我封装了存储管理工具做定时清理和容量控制。上报时机上优先使用 requestIdleCallback 在浏览器空闲时上报，不阻塞主线程；页面卸载时改用 navigator.sendBeacon 保证数据必达，避免 fetch 或 xhr 被浏览器中断，大幅提升上报成功率。
>
>第三，实现了全场景错误捕获。一方面通过 window.onerror、unhandledrejection 捕获全局同步、异步错误；另一方面做了框架深度适配，在 Vue 中通过 app.config.errorHandler 捕获全局错误，用 onErrorCaptured 捕获组件级错误，同时也兼容了 React 的错误捕获逻辑，保证框架层面错误不遗漏。
>
>第四，补充了网络请求异常采集。通过监听 unhandledrejection 等事件，自动捕获接口请求异常，结合状态码、接口地址、耗时等信息，形成完整的异常上下文，方便快速定位问题。
>
>整体 SDK 追求轻量、低侵入、高可靠，在核心监控能力上接近成熟方案，同时更贴合业务场景，体积更小、数据更可控。

### 8、框架原理

>我有看过 Vue3 的源码，它的核心实现可以分成三个部分：**编译时、运行时、响应式系统**。
>
>1. **编译时（Compiler）**：
>
>   它的核心目标是把我们写的 `.vue` 单文件组件和 `template` 模板，编译成浏览器能执行的 JavaScript 代码。
>
>   整个过程是：`compiler-sfc` 先解析 `.vue` 文件，把 template、script、style 三个部分分离出来；然后 `compiler-core` 会对模板进行词法分析、语法分析，生成 AST（抽象语法树），再把它编译成高效的渲染函数；最后 `compiler-dom` 会做一些浏览器端的 DOM 适配，比如处理指令、事件等。
>
>   
>
>2. **运行时（Runtime）**：
>
>   运行时的核心是 `render` 渲染函数和 `patch` 算法。编译生成的渲染函数会生成虚拟 DOM，然后运行时会根据虚拟 DOM，高效地更新真实 DOM。
>
>   它通过 `patch` 算法对比新旧虚拟 DOM 的差异，只更新变化的部分，而不是重新渲染整个页面，这就是 Vue 高效更新的核心。
>
>   
>
>3. **响应式系统**：
>
>   这是 Vue 最核心的特性。Vue2 是用 `Object.defineProperty` 实现的，但它有缺陷，比如无法监听对象新增 / 删除属性，也不能监听数组索引变化。
>
>   Vue3 改用 `Proxy` 代理整个对象，能监听所有属性的访问和修改，包括新增、删除，也能监听数组变化。
>
>   当数据被访问时，会通过 `get` 拦截器收集依赖；当数据被修改时，会通过 `set` 拦截器触发更新，通知所有依赖它的组件重新渲染
>
>- 挂载流程
>
>  Vue3 的 `createApp` 和 `mount` 挂载流程，本质上是「**创建应用实例 → 编译模板 → 生成虚拟 DOM → 渲染真实 DOM**」的完整过程，核心步骤是这样的：
>
>  1. **`createApp(App)`：创建应用实例**
>
>     这一步会创建一个根应用实例，它会接收我们的根组件 `App.vue` 作为参数，同时初始化应用的配置、插件、全局组件等上下文，返回一个应用实例对象。
>
>     
>
>  2. **`app.mount('#app')`：开始挂载流程**
>
>     当我们调用 `mount` 方法并传入 DOM 容器（比如 `#app`）时，整个渲染流程正式启动：
>
>     
>
>     - **编译阶段**：首先会对 `App.vue` 组件进行解析。如果是运行时编译模式，会把 `template` 模板解析成 AST（抽象语法树），再编译成可执行的 `render` 渲染函数；如果是构建时预编译，直接使用打包生成的渲染函数即可。
>     - **生成虚拟 DOM**：执行 `render` 函数，生成根组件的虚拟 DOM 树（VNode），它是真实 DOM 的轻量级描述对象。
>     - **首次渲染（patch）**：运行时的 `patch` 算法会把虚拟 DOM 转换成真实 DOM 节点，并一次性插入到 `#app` 容器中。
>     - **响应式绑定**：同时会建立数据与视图的响应式关联，后续数据变化时，会重新生成新的虚拟 DOM，通过 `patch` 算法对比差异，局部更新真实 DOM。

### 9、react /  vue

>我理解的 Vue 和 React，都是主流的前端框架，但在设计理念和开发体验上有很明显的差异，核心区别可以从这几个维度来看：
>
>1. **设计理念**
>
>   - Vue 是**渐进式框架**，官方提供了完整的全家桶方案（Vue Router、Pinia 等），开箱即用，上手门槛低，能快速搭建项目；同时也支持按需引入，适合从简单到复杂的项目。
>   - React 是**函数式、声明式的 UI 库**，核心只关注视图层，生态更偏向社区驱动，状态管理、路由等工具需要自己选型组合，灵活性更高，更适合大型团队和复杂项目。
>
>   
>
>2. **数据更新机制**
>
>   - Vue 基于**响应式系统**（Vue2 的 `Object.defineProperty` / Vue3 的 `Proxy`），数据是可变的，直接修改数据就能触发更新，比如 `count++`，框架内部会自动做依赖收集和批量更新，开发体验更直观。
>   - React 是**单向数据流**，数据是不可变的，必须通过 `setState` 或 `useState` 返回的更新函数来修改数据，比如 `setCount(prev => prev + 1)`，这种方式让数据变更可追踪，更利于调试和维护大型项目。
>
>   
>
>3. **模板与渲染方式**
>
>   - Vue 支持**模板 + JSX 双模式**，默认的 HTML 模板写法更贴近原生，学习成本低，同时也支持 JSX 满足复杂场景需求。
>   - React 采用**纯 JSX**，所有逻辑都在 JavaScript 中实现，逻辑和视图完全融合，灵活性更强，也更利于做类型检查和工具链优化。
>
>   
>
>4. **适用场景**
>
>   - Vue 更适合需要快速交付、追求开发效率的项目，尤其是中小团队和中小型项目，开箱即用的方案能降低决策成本；同时 Vue3 也完全能支撑大型复杂项目，比如 B 站、饿了么都在用。
>   - React 更适合大型团队、需要高度定制和长期维护的项目，单向数据流和函数式的设计，更利于团队协作和代码规范统一，生态也更成熟，有更多企业级解决方案。
>
>   
>
>简单来说，Vue 更注重开发效率和上手体验，React 更注重灵活性和工程化能力，两者没有绝对的优劣，只是不同设计理念下的选择。
