### 核心难点包装（深挖准备）

你要把亮点从“我用了 WebWorker”转变为“**我解决了流式传输下的渲染抖动与语法完整性问题**”。

#### 难点 1：流式 Markdown 的“闪烁/抖动”与语法截断问题（这是杀手锏）

**场景描述：**
大模型吐字是流式的。假设大模型要输出一个代码块，它是一个字符一个字符吐出来的：

* T1: ``` (页面显示：反引号)
* T2: ``` (页面显示：两个反引号)
* T3: ``` (页面显示：三个反引号，可能突然变成代码块样式，或者还没变)
* T4: `j` -> `ja` -> `jav` ...

在这个过程中，如果每接收一个字符就 parse 一次，HTML 结构会剧烈变化（从普通文本瞬间变成 `<pre><code>`），导致页面高度跳动，用户视觉上会看到**内容闪烁**。

**高分解决方案（你可以这样吹）：**

> “在处理流式 Markdown 时，最大的痛点是 **Token 截断（Incomplete Syntax）**。为了解决这个问题，我没有简单的收到一个 chunk 就渲染一次，而是做了一个 **‘脏解析缓冲层’**：
> 1. **防抖渲染（Throttle）：** 不会来一个字符就 parse 一次，而是设定了 50ms-100ms 的阈值，累计一定字符量再解析，减少主线程压力。
> 2. **末端修补（关键点）：** 在流没结束前，如果检测到末尾有**未闭合的 Markdown 语法**（比如只有半个 `**` 加粗，或者未闭合的 ```` ` 代码块），我会通过正则表达式或 AST 分析，在渲染前暂时**补全闭合标签**，直到收到换行符或完整闭合标记。这避免了页面因为 HTML 结构频繁突变导致的‘布局抖动’（Layout Thrashing）。”
> 
> 

#### 难点 2：代码块的高性能处理（回答面试官的追问）

**场景描述：**
代码块需要高亮（Syntax Highlighting），这通常很慢。

**高分解决方案：**

> “关于代码块，我做了特殊优化。我使用了 `markdown-it` 配合 `highlight.js`。
> 1. **按需加载语言包：** `highlight.js` 全量包太大。我配置了动态 import，只有当大模型输出 ````python` 时，我才去异步加载 Python 的高亮语法包，减少首屏体积。
> 2. **自定义 Renderer：** 我重写了 `markdown-it` 的 renderer rule。在解析出的 `<pre>` 标签上，我不仅注入了高亮 HTML，还**直接在 AST 转换阶段注入了 Metadata**（如语言类型、原始文本），方便后续组件层直接读取这些属性来实现‘一键复制’功能，而不需要再次从 DOM 中提取文本，保证了数据流的清晰。”
> 
> 

#### 难点 3：超长对话的虚拟列表 + 动态高度（你已经准备了，但要结合 AI 场景）

**场景描述：**
AI 回答可能很长，也可能很短，且高度是动态变化的（因为在流式生成）。虚拟列表通常需要固定高度。

**高分解决方案：**

> “虚拟列表结合流式输出是一个难点。
> 1. **锚点锁定：** 当用户正在看最新的消息（接近底部）时，我会利用 `ResizeObserver` 监听当前正在生成的那个 DOM 节点的高度变化，并自动滚动到底部。
> 2. **视口锁定：** 如果用户向上滚动回顾历史消息，我会暂停自动滚动逻辑（Auto-Scroll Lock），防止新内容生成把用户当前的视口顶走。
> 3. **高度预估：** 对于历史消息，我利用已渲染的真实高度缓存；对于正在生成的流式消息，我暂时不通过虚拟列表卸载它，直到该条消息 `Status: Completed`，再将其纳入虚拟列表的管理范围。”
>
