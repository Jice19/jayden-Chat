## 不同域之间共享数据

a.baidu.com和b.baidu.com之间不能共享localstorage中的数据，同源策略，二者主域名相同baidu.com,但是子域名不同，属于不同源

如果想要实现共享数据，可以使用以下几种通信方式

| 方法                      | 核心逻辑                                                     | 适用场景                     |
| ------------------------- | ------------------------------------------------------------ | ---------------------------- |
| 1. `document.domain`      | 两页面均设置 `document.domain = "主域名"`（如`.baidu.com`），统一域上下文 | 同主域、同协议端口，简单场景 |
| 2. `iframe + postMessage` | 主页面嵌入目标子域名 iframe，通过`postMessage`安全传递数据，目标页读写 localStorage | 通用跨子域，安全性高（推荐） |
| 3. Cookie 中转            | 设置 Cookie 的`domain=.主域名`，通过 Cookie 传递少量数据（≤4KB），`document.cookie = "key=val; domain=.baidu.com; path=/"` | 少量数据、兼容老浏览器       |



| 方法                   | 是否要求主域名相同 | 跨域范围                                    |
| ---------------------- | ------------------ | ------------------------------------------- |
| `document.domain`      | 是                 | “权限解锁方案”—— 解锁子域名间的直接访问权限 |
| Cookie 中转            | 是（共享场景）     | a.baidu.com可以拿到b.baidu.com的cookie信息  |
| `iframe + postMessage` | 否                 | 所有跨域场景（含不同主域）                  |



### iframe通信关键点 

1. **嵌入限制**：目标页需解除`X-Frame-Options`限制（服务器配置`ALLOW-FROM 主域名`/`SAMEORIGIN`），否则无法嵌入；
2. **协议端口**：主页面与 iframe 目标页必须协议（均 HTTP / 均 HTTPS）、端口一致，避免混合内容拦截；
3. 通信安全：
   - 发送`postMessage`时精准指定目标域名，禁止用`*`；
   - 接收消息必须验证`e.origin`，拒绝非可信来源；
4. **加载时机**：需监听 iframe 的`load`事件，确保目标页加载完成后再通过`postMessage`通信；
5. **访问限制**：主页面无法直接访问 iframe 的`localStorage`，必须通过`postMessage`中转，不可直接跨域操作；
6. **样式配置**：数据共享场景建议隐藏 iframe（`display: none; width:0; height:0`），不占用页面空间。

```
<!-- a.baidu.com 主页面 -->
<!DOCTYPE html>
<html>

### 插件执行

每个插件存在不同生命周期，分成前置与后置阶段

```
//vite.config.js
// 测试插件 A（模拟 vue 插件）
const pluginA = {
  name: 'plugin-A',
  // 前置阶段钩子（正序执行）
  buildStart() {
    console.log('✅ [前置阶段] plugin-A: buildStart 执行')
  },
  resolveId(id) {
    console.log('✅ [前置阶段] plugin-A: resolveId 执行 →', id)
    return null // 不修改路径，仅打印
  },
  load(id) {
    console.log('✅ [前置阶段] plugin-A: load 执行 →', id)
    return null // 不加载代码，仅打印
  },
  // 后置阶段钩子（倒序执行）
  transform(code) {
    console.log('🔄 [后置阶段] plugin-A: transform 执行')
    return code // 不修改代码，仅打印
  },
  generateBundle(options, bundle) {
    console.log('🔄 [后置阶段] plugin-A: generateBundle 执行')
  },
  writeBundle() {
    console.log('🔄 [后置阶段] plugin-A: writeBundle 执行')
  }
}

export default definConfig({
	plugins:[
		pluginA,
		pluginB,
		pluginC
	]
})
```



**前置阶段（buildStart/resolveId/load）**：严格按照 `plugins` 数组顺序（A→B→C）执行，核心是 “读取 / 解析代码”；

**后置阶段（transform/generateBundle/writeBundle）**：严格按照数组倒序（C→B→A）执行，核心是 “修改 / 生成产物”；

#### 总结

1. 同一个插件会同时包含**前置 + 后置**钩子，不是分开的两个插件；
2. 前置钩子正序执行（数组从头到尾），后置钩子倒序执行（数组从尾到头）；
3. 配置 Sentry 时，要把它放在压缩插件**前面**（数组中），才能让压缩先执行、Sentry 后处理。

## 函数式编程思想

函数式编程思想最大的特点是不可变性、无副作用（不修改全局变量和外部函数）以及纯函数（输出值只和输入值有关，无副作用）

1、纯函数，比如数组的slice方法和splice方法

slice：纯函数，输出指定数据，左闭右开

splice：不是纯函数，array.splice是将指定数据切割舍弃

### 纯函数的特点

（1）可缓存：由于输入值与输出值一一对应，所以可以缓存已经输出过的数据

3、函数柯里化

```
function curry(fn){
	function curried(...args){
		if(args.length >= fn.length){
			return fn.apply(this,args)
		}else{
			return function(...moreArgs){
				return curried.apply(this,args.concat(moreArgs))
			}
		}
	}
	return curried
}
```

4、compose函数，实现将多个函数组合成一个新函数，每个函数的输出作为下一个函数接收的参数

function compose(...fns){
    return function(x){
        return fns.ruduceRight((value,fn) =>fn(value),x)
    }
}

## 计网

### OSI七层模型：

1、物理层：直接和物理介质打交道的，传输**比特流**

2、数据链路层：把比特流拼接成**数据帧**

3、网络层：寻址（IP作为媒介）+  路由

4、传输层：定义端口号，拥有TCP （三次握手四次挥手） UDP（速度快但是不稳定）

5、会话层（报文）：类似电话通话，在发送方和接收方进行通信时创建、维持的地方

6、表示层：解码

7、应用层：例如ajax调用接口发送http请求

### TCP|IP  五层|实际是四层模型

五层模型： 物理层  数据链路层  网络层  传输层  应用层

四层模型： 网络接口层  网际联络层  传输层  应用层



TCP & UDP 都属于OSI的传输层，

TCP面向连接，UDP无连接

TCP传输字节流，UDP面向报文

TCP安全传输慢 ，UDP尽快传输但是不保证交付给对方

TCP全双工通信，UDP支持一对一、一对多、多对一和多对多交互通信

### 三次握手

tcp是面向连接的：

SYN:发起新连接       ACK：确认序列号有效      seq:随机生成的序列号   ack：确认号 = 对方 seq + 1

### 四次挥手

![75457434686](assets/1754574346861.png)



**1、**客户端向服务端发送FIN包，并携带随机数序列号进入wait1阶段，  **2、**服务端接收之后返回ACK标识，并返回序列号+1，客户端进入进入wait2阶段（期间处理未完成的请求之类的），处理完之后  **3、**服务端才会进行第三次挥手，客户端接收到第三次挥手的包之后进入超时等待状态（持续几分钟，如果第四次挥手返回的ACK丢失的话，这段期间可以处理好服务端再次发送的第三次挥手包），**4、**最后客户端发送第四次挥手发送ACK标记

### GET和POST的区别

**本质**上都是`TCP链接`，并无差别

区别：

1. GET在浏览器回退时时无害的，而POST会再次提交请求

2. GET请求会被浏览器主动Cache，而POST不会，除非手动设置（主要由服务端的响应头决定，服务端设置：

   ```
   app.post('/api/data', (req, res) => {
     // 处理 POST 请求逻辑...
     
     // 设置响应头允许缓存
     res.setHeader('Cache-Control', 'public, max-age=3600');
     res.setHeader('Expires', new Date(Date.now() + 3600 * 1000).toUTCString());
     res.setHeader('ETag', 'data-version-1'); // 可根据数据版本动态生成
     
     res.json({ data: '缓存的 POST 响应' });
   });
   ```

3. GET请求的参数可以完整被存放在浏览器历史记录里，但是POST不可以存放在浏览器历史记录里面

4. GET没有POST安全，因为请求参数暴露在url上，不能用来传递敏感信息

5. GET通过URL传参，POST放在Request body里面

3、精炼解释原型链



**“核心回答”：** 原型链的最大好处是提升内存效率；允许成千上万个对象实例共享同一套定义在单个原型对象上的公共方法，从而避免每个实例重复创建每个实例方法



**“拓展解释”：**举个例子，当我们使用Array创建一个数组实例的时候，不需要去给每个数组对象挂载方法（如.map,.filter等等），因为这些数组实例都通过原型链，共同指向并使用定义在Array.prototype这个全局唯一的原型对象上的那一套方法

//广度优先遍历
function bfs(graph,start){
	const visited = new Set()
	visited.add(start)
	const queue = [start]  //将首节点放入队列
	
	const result = []  //接收已访问的节点
	while(queue.length>0){
		const node = queue.shift() //移除队首元素
		result.push(node)
		
		for(const neighbor of graph[node]){
			if(!visited.has(neighbor)){
				visited.add(neighbor)
				queue.push(neighbor)
			}
		}
	}
	return result
}

### 快慢指针找链表的相遇点和环起点

相遇点：同时从起点出发，快指针走两步，满指针走一步，这次相遇就是相遇点

环起点：慢指针从起点出发，快指针从相遇点出发，都只走一步，下一次相遇就是环的起点

![image-20251203091617297](assets/image-20251203091617297.png)



相遇点二者走的路程：慢指针：L+X         快指针： 2(L+X)  

推导公式：2(L+X)= L+X+nC

​				    L+X = nC  =》 L = nC - X  =》 L = (n-1)C + (C-X) =》 L = (n-1)C + Y  =》**起点到环起点的距离 `L`，等于 “相遇点到环起点的距离 `Y`” 加上 “环长度的 `(n-1)` 倍”**。

## npm  pnpm yarn

三者都是处理依赖管理的包管理工具，主要区别是在磁盘存储占用、解析速度、锁文件机制和生态兼容等方面差异显著

npm、pnpm、yarn 作为 Node.js 主流包管理器，核心目标一致但各有侧重，核心总结如下：

### 1. 核心差异核心

| 维度         | 核心结论                                                     |
| ------------ | ------------------------------------------------------------ |
| 性能 / 速度  | pnpm 最优（硬链接复用 + 并行安装），yarn 次之，npm（v7+）最慢 |
| 磁盘占用     | pnpm 极致节省（全局存储池 + 硬链接），yarn 中等，npm 最高（重复存储依赖） |

| 兼容性       | npm 最好（官方默认），yarn 优秀，pnpm 良好（极少数依赖 hoist 的包需兼容） |

### 2. 适用场景速选

## watch处理深度监听

在 Vue 中 `watch` 处理深度监听（监听对象 / 数组的深层属性变化），核心是通过配置 `deep: true` 实现，同时 Vue2 和 Vue3 的写法基本一致，仅语法细节略有差异，我分点说明：

###  核心配置：`deep: true`

当监听的目标是**对象 / 数组**（而非简单类型），默认 `watch` 只监听 “引用地址变化”（比如直接替换整个对象），不会监听内部属性变化；添加 `deep: true` 后，会递归监听对象所有层级的属性变化。



###  优化：精准监听对象单个深层属性（避免全量深度监听）

`deep: true` 会递归监听整个对象，若对象层级深、属性多，可能有性能开销；可通过 “字符串路径” 直接监听单个深层属性，无需开启 `deep: true`，更高效：

###  关键注意事项：

- **数组监听**：数组的 `push/pop/splice` 等方法会触发 `watch`（无需 `deep: true`），但通过 “下标修改元素”（如 `arr[0] = 1`），Vue2 需 `deep: true`，Vue3 因 `Proxy` 可直接监听；
- **性能考量**：全量深度监听（`deep: true`）尽量用在小对象上，大对象优先用 “精准监听单个属性”；
- **引用类型注意**：若直接替换整个对象（如 `user.value = { ... }`），即使不开启 `deep: true`，也会触发 `watch`（因为引用地址变了）。

### 替代方案

同时也可以使用computed去监听具体的属性，提取需要深度监听的属性，computed会响应依赖变化

`watchEffect` 是 Vue3 新增的监听方式，无需显式指定监听目标，会自动追踪回调中用到的**所有响应式依赖**（包括深层属性），天然支持深度监听：

## flex核心进阶

flex：1是三个属性的整合 flex-grow flex-shrink  flex-basis （1 1 0%）自动放大、缩小、自适应占满

**flex-grow:** 仅在父元素声明display:flex生效，用来放大指定元素，平分`总宽度-基准宽度`然后按**比例进行平分**

```css
.flex-container { display: flex; width: 500px; }
.item1 { width: 100px; flex-grow: 1; }
.item2 { width: 200px; flex-grow: 1; }
```

- 基准尺寸（flex-basis:auto）= `width` → 总基准 = 100+200=300px；

- 剩余空间 = 500-300=200px；

- `flex-grow:1` 均分剩余空间 → 各分 100px；

- 最终宽度：item1=100+100=200px，item2=200+100=300px；

  →此时 `width` 决定基准，`flex-grow` 基于 `width` 瓜分剩余空间

  

**flex-shrink:** 缩小元素，

```css
.container { display: flex; width: 300px; }
.item1 { width: 200px; flex-shrink: 1; }
.item2 { width: 200px; flex-shrink: 1; }
```

- 基准尺寸（`flex-basis: auto`）= `width` → 总基准 = 200+200=400px；

- 需缩小的空间 = 400-300=100px；

- 缩小权重 = `width × flex-shrink` → item1: 200×1=200；item2: 200×1=200；

- 每个项目缩小的尺寸 = 总缩小空间 ×（自身权重 / 总权重）→ 100 × (200/400) = 50px；

- 最终宽度：item1=200-50=150px；item2=200-50=150px；

  →此时 `width` 是缩小的基准，`flex-shrink` 基于 `width` 计算缩小比例

  
  
  flex：0 （0 1 auto） 保持自身宽度，不放大抢占空间，空间不足自动缩放
  
  

**flex-basis**:  元素的基准宽度（`关于主轴上的`），如果为auto的时候，值为width，如果设有数值的话，优先级高于`width`



## type of /instance of

### 3. type of核心作用

- 快速判断**基本数据类型**（除 `null`）：比如判断变量是否为 `undefined`、是否为数字 / 字符串；
- 区分 “函数” 和 “普通对象”：`typeof fn === "function"` 是可靠的判断方式。

### 4. 局限性

- 无法区分 `null`、普通对象、数组（均返回 `"object"`）；
- 无法判断引用类型的具体细分（比如分不清数组和普通对象）。



### instance of

- 精准判断**引用类型的具体类型**：比如区分数组和普通对象（`arr instanceof Array`）；
- 验证类实例、自定义构造函数的实例（比如判断是否为 `Promise` 实例：`p instanceof Promise`）；
- 检测原型链继承关系（比如判断是否继承自某个父类）。

#### 4. 局限性

- 无法判断基本数据类型（比如 `123 instanceof Number` → `false`，但 `new Number(123) instanceof Number` → `true`）；
- 原型链被修改后会失效：比如 `Object.setPrototypeOf(arr, null)` 后，`arr instanceof Array` → `false`；
- 跨窗口 / 跨框架时，因构造函数的原型不共享，可能判断错误（比如 iframe 中的数组 `instanceof Array` → `false`）。



## SSE与Fetch

sse是基于http的现代api，可以实现服务端主动向客户端推送数据，但是只支持推送文本数据，如果想要传输图片需要使用base64解码或者使用url进行传递，减少了轮询实现双向通信导致的服务端压力过大的问题

### 面试标准答案：【fetch 流式和 SSE 的核心区别】（完整话术，无任何错误，背完直接答）

> 面试官您好，fetch 流式响应和 SSE 都能实现打字机效果，两者的核心不是「长连接和短连接」的区别，而是「都是 HTTP 长连接，但连接的生命周期和使用方式不同」，具体区别如下：

1. 连接本质：两者都是 HTTP 长连接，在控制台都只会看到 1 个请求连接，区别在于

   生命周期：

   - fetch 流式是「一次性长连接」：服务端推完本次请求的所有数据后，连接自动关闭，是用完即断的任务型连接；
   - SSE 是「永久长连接」：连接建立后会一直保持打开，除非手动关闭，是常驻型连接；

2. 数据推送：

   - fetch 流式只能通过当前连接，推送「本次请求的单次数据流」，多轮请求需要重新发起新的 fetch 请求；
   - SSE 可以通过同一个常驻连接，推送「多次、任意的数据流」，多轮请求无需重新发起，连接复用；

3. 适用场景：

   - 单轮的流式请求（比如单轮 AI 回答、大文件下载），用 fetch 流式，轻量无残留；
   - 持续的流式推送（比如多轮 AI 对话、实时日志），用 SSE，连接复用更高效；

4. 总结：两者都是长连接，核心差异是「连接是否常驻、是否支持复用」。

### sse兼容性以及解决方案

sse兼容现在95％的浏览器，但是低版本浏览器以及ie全系不支持

解决方案：长轮询、

使用成熟的 SSE 兼容库（如 `eventsource-polyfill`），自动为不支持的浏览器模拟 `EventSource` API，底层仍基于长轮询实现：



### 降级方案：长轮询（兼容性+易用性）

“请求挂起 + 持续监听响应+ 接收响应之后重连”

#### 优点

1. **实时性较高**：服务端有数据时能立即推送，延迟远低于短轮询。
2. **兼容性好**：基于标准 HTTP 协议，无需额外插件，支持所有浏览器和网络环境（包括防火墙、代理服务器）。
3. **资源消耗可控**：相比短轮询的 “高频无效请求”，长轮询大幅减少了请求数和网络传输开销；相比 WebSocket，服务端实现更简单。

#### 缺点

1. **服务器资源占用**：大量客户端的长连接会占用服务端的文件描述符，高并发场景下可能引发连接数瓶颈。
2. **头部开销大**：每次请求都携带 HTTP 头部信息，相比 WebSocket 的帧传输，存在额外的带宽消耗。
3. **无法真正双向通信**：本质是客户端主动拉取，服务端被动响应，不支持服务端主动向客户端发起消息（需客户端先发起请求）。

### web worker

**先算文件 hash 定 “身份”，再逐片算 MD5 保 “完整”，前者是前提，后者是细节，两者配合实现 “全局标识 + 局部校验” 的完整体系**。

- [x] 项目中webworker方面

JS 是单线程模型，主线程同时负责「DOM 渲染、用户交互、业务逻辑执行」，如果在主线程处理**大文件分片、MD5 加密这类计算密集型任务**，会阻塞主线程的事件循环，导致页面 UI 卡顿、点击无响应甚至浏览器提示 “页面未响应”，这也是 Web Worker 的核心解决场景。

在大文件上传场景中，我会这样落地 Web Worker：

1. **创建独立的 Worker 线程**：新建`upload.worker.js`文件，把文件分片、MD5 加密的核心逻辑抽离到这个子线程中，主线程仅负责触发上传、接收结果、更新 UI；
2. **线程间通信**：主线程通过`worker.postMessage()`把文件对象、分片大小等参数传给 Worker，Worker 内部通过`self.onmessage`接收参数；加密完成后，Worker 再通过`self.postMessage()`（或`this.postMessage()`，两者等价）把分片后的 MD5 值、文件片段传回主线程；
3. **优化通信效率**：我在 Excel 大文件上传中，优先采用结构化克隆算法传递 File 对象 —— 浏览器对 File/Blob 的结构化克隆仅共享二进制引用，实现零拷贝传输，同时主线程保留文件访问权，可灵活控制上传流程（如取消、断点续传）；仅在处理 500MB 以上超大文件时，才会将 File 转为 ArrayBuffer 并通过可转移对象转移所有权，平衡性能和灵活性
4. **资源回收**：分片加密完成后，主线程调用`worker.terminate()`销毁 Worker，避免内存泄漏。

另外补充：

> Worker 线程中没有`window`对象，`this`和`self`都指向 Worker 的全局对象`DedicatedWorkerGlobalScope`，无法访问 DOM、`alert`等主线程 API，仅能执行纯计算逻辑。
>
> 默认情况下`postMessage`采用「结构化克隆」机制（复制数据），大文件分片的`ArrayBuffer`数据拷贝会产生大量内存开销，因此我会使用**Transferable Objects（可转移对象）** 实现 “零拷贝”—— 把数据的内存所有权从一个线程转移到另一个线程（类似 “剪切粘贴”），避免拷贝损耗，比如传递分片的`ArrayBuffer`时，在`postMessage`第二个参数声明可转移对象：`postMessage({ buffer }, [buffer])`；

- [x] Web Worker 有哪些使用限制？（必考）

**回答**：

Web Worker 的核心限制集中在「环境隔离」和「通信约束」，主要有 4 点：

1. **DOM 访问限制**：Worker 线程中没有`window`/`document`/`body`等对象，无法操作 DOM、调用`alert`/`confirm`等浏览器 UI API；
2. **通信限制**：只能和创建它的主线程通信，无法跨 Worker 通信（多 Worker 通信需通过主线程中转）；
3. **资源限制**：不能读取本地文件（如`file://`协议），只能加载同源脚本，无法访问`localStorage`（可通过`IndexedDB`实现数据持久化）；
4. **生命周期限制**：Worker 是 “一次性” 的，任务完成后需手动`terminate()`销毁，否则会一直占用线程资源。





- [x] 用了 Transferable Objects “零拷贝” 后，主线程还能访问原数据吗？

**回答**：

不能。「零拷贝」的本质是**内存所有权转移**，数据一旦从主线程转移到 Worker 线程，主线程的原数据会被清空（变为`null`），反之亦然。

比如主线程传递`const buffer = new ArrayBuffer(1024)`给 Worker，并用`postMessage({ buffer }, [buffer])`声明转移后，主线程的`buffer`会变成`null`，只有 Worker 能访问该内存。

如果需要主线程和 Worker 共享数据（而非转移），可以用`SharedArrayBuffer`（需配置跨域头`Cross-Origin-Opener-Policy`和`Cross-Origin-Embedder-Policy`）

## ajax和fetch

1. 本质差异：Ajax 是基于 `XMLHttpRequest` 的传统异步请求方案，Fetch 是现代 Promise 式异步请求 API，设计更优雅；
2. 核心优势：Fetch 支持 Promise、链式调用、流式处理，代码更简洁；Ajax 兼容性更好、支持进度监控，使用更成熟；
3. 注意点：二者默认都不携带 Cookie，Fetch 需手动处理 HTTP 错误和响应转换，Ajax 需避免回调地狱；
4. 实际开发：现代项目优先使用 Fetch（或封装后的 Axios，基于 `XMLHttpRequest` 封装，兼顾 Fetch 优雅性和 Ajax 功能性），老旧项目保留 Ajax。





| 对比维度         | Ajax（XMLHttpRequest）                                       | Fetch API                                                    |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **API 设计风格** | 传统回调式 API，基于构造函数实例化，接口冗余杂乱（如 `open()`/`send()` 分步调用） | 现代 Promise 式 API，基于函数调用，语法简洁直观，支持链式调用 |
| **回调方式**     | 依赖回调函数（`onreadystatechange`/`onload`），易产生 “回调地狱” | 基于 Promise，支持 `then/catch` 链式调用，也可配合 `async/await` 解决回调地狱 |
| **默认请求配置** | 默认不会携带 Cookie，需手动设置 `xhr.withCredentials = true` | 默认也不携带 Cookie，需通过 `credentials` 配置（`same-origin`/`include`/`omit`） |
| **错误处理机制** | 仅在网络层面错误（如断网）时触发 `onerror`，HTTP 状态码 4xx/5xx 不会视为错误（需手动判断 `status`） | 仅在网络错误或请求被中止时会拒绝 Promise，HTTP 4xx/5xx 仍会视为成功响应（需手动通过 `response.ok` 判断） |
| **响应处理**     | 直接返回响应数据，可按需获取 `responseText`/`responseXML`，无需额外转换 | 返回一个 `Response` 对象，需通过 `json()`/`text()`/`blob()` 等方法手动转换为对应数据格式（异步操作） |
| **中断请求**     | 通过 `xhr.abort()` 方法可直接中断请求，使用简单              | 需要通过 `AbortController` + `AbortSignal` 实现中断，相对复杂 |
| **进度监控**     | 支持通过 `onprogress` 事件监控请求 / 响应进度（如文件上传进度） | 原生不支持进度监控，需借助 `ReadableStream` 实现，复杂度较高 |
| **浏览器兼容性** | 兼容性极佳，支持所有现代浏览器及老旧浏览器（如 IE6+）        | 支持现代浏览器，不兼容 IE 浏览器，需通过 polyfill 兼容老旧环境 |
| **流式处理**     | 不支持流式响应，需等待完整响应后才能处理数据                 | 支持 `ReadableStream` 流式处理，可逐步读取响应数据（适合大文件处理） |

###  五、面试标准答案：axios/fetch/ajax 的区别（完整话术，背完直接答，分点清晰，逻辑满分）

> 面试官您好，axios、fetch、ajax 三者的区别，核心是「层级关系 + 本质 + 特性」的区别，我从三个层面说明：

#### 1. 首先是三者的本质和层级关系（定调）

- AJAX 是一种**异步请求的技术思想**，不是具体的 API；实现这个思想的浏览器原生底层 API 有两个：`XMLHttpRequest(XHR)` 和 `fetch`，这两个 API 是平级的，都是浏览器原生能力。
- 我们常说的「原生 AJAX」就是指直接调用 `XMLHttpRequest`，它是老牌的原生请求 API；fetch 是 ES6 推出的新一代原生请求 API，和 XHR 平级，不是对 AJAX 的封装。
- axios 是一个**基于原生 XHR 封装的第三方请求库**，不是浏览器原生 API，是为了解决原生请求的痛点而生的。

#### 2. 其次是三者的核心特性区别

#### ① 原生 AJAX (XHR)

是最底层的原生 API，兼容性无敌，但 API 设计繁琐，基于回调函数，容易产生回调地狱，没有拦截器、超时处理等实用功能，开发成本极高，实际开发中几乎不会手写原生 XHR。

#### ② 原生 fetch

是新一代原生 API，基于 Promise 设计，解决了回调地狱问题，语法更简洁，还支持流式响应。但它有很多反人类的原生缺陷：不会自动捕获 HTTP 错误、不会自动解析 JSON、没有拦截器、没有超时配置，默认不携带 Cookie，这些缺陷导致原生 fetch 必须封装后才能使用，否则开发体验极差。

#### ③ axios

是基于 XHR 封装的第三方库，整合了原生请求的所有优点，规避了所有缺陷。它基于 Promise 设计，支持 async/await，开箱即用：自动捕获 HTTP 错误、自动解析 JSON、内置请求 / 响应拦截器、超时处理、请求取消、并发请求等所有刚需功能，兼容性好，开发成本极低，是目前前端项目开发的首选。

#### 3. 最后补充一个核心区别（加分项）

- axios 是基于 XHR 封装的，不是基于 fetch；因为 fetch 的底层缺陷无法被完美封装解决，而 XHR 的所有缺陷都能通过封装规避。
- 实际开发中：项目开发用 axios，原生开发 / 轻量场景可以用封装后的 fetch，几乎不会手写原生 XHR。

------

#### ✅ 最后补充：开发中的实际使用场景

1. 企业级项目开发 → **首选 axios**：功能全、稳定、开发效率高；
2. 轻量 H5 / 小程序 / 无构建工具的项目 → 封装后的 fetch：体积小，无需引入第三方库；
3. 兼容 IE 等低版本浏览器 → 必须用 axios（原生 fetch 不兼容 IE）；
4. 处理大文件上传 / 下载等特殊场景 → 可以用 fetch 的流式响应能力。

## 1. LocalStorage + storage 事件

> 原理

- **LocalStorage**：浏览器本地存储（同一域名下的标签页共享数据）；
- **storage 事件**：当某一标签页修改 LocalStorage 时，其他标签页会触发 `storage` 事件，从而感知数据变化。

> 代码示例

```
// 标签页 A（发送数据）
localStorage.setItem('msg', '来自标签页A的消息');

// 标签页 B（接收数据）
window.addEventListener('storage', (e) => {
  if (e.key === 'msg') { // 只监听指定 key 的变化
    console.log('收到新消息：', e.newValue); // e.newValue 是新值，e.oldValue 是旧值
  }
});
```

> 特点

✅ 优点：API 简单，无需额外依赖；

❌ 缺点：

- 同一标签页修改 LocalStorage 不会触发自身的 `storage` 事件（仅其他标签页触发）；
- 只能传递字符串（复杂数据需 `JSON.stringify`/`JSON.parse`）；
- 数据存在本地，可能被篡改。

## 2. BroadcastChannel

> 原理

浏览器原生 API，专门用于**同一域名下的标签页 /iframe 之间广播通信**，直接发送任意类型数据（无需序列化）。

代码示例

```
// 标签页 A（创建频道并发送消息）
const channel = new BroadcastChannel('my-channel'); // 频道名一致才能通信
channel.postMessage({ type: 'greet', content: 'Hello 其他标签页' });

// 标签页 B（监听同一频道的消息）
const channel = new BroadcastChannel('my-channel');
channel.onmessage = (e) => {
  console.log('收到广播：', e.data); // e.data 是发送的原始数据
};

// 关闭频道（可选）
channel.close();
```

> 特点

✅ 优点：

- 支持任意数据类型（对象、数组等），无需序列化；

- 同一标签页也能接收自己发送的消息；

- API 语义化，专门用于跨页通信；

  

  ❌ 缺点：

- 兼容性：IE/Edge 18 以下不支持；

- 仅同一域名下可用（同源策略）。

## vue源码解析

###  一、vue和React设计哲学

①Vue 基于**全局化的自动响应式机制**实现视图更新，React 则是基于**手动触发的细粒度更新模式**完成渲染刷新；

②二者核心优势各有侧重且十分鲜明：Vue 凭借自动响应式让前端开发的**心智负担大幅降低**，开发流程更简洁、上手更轻松，极大简化前端开发落地效率；React 依托手动可控的更新逻辑，让复杂业务场景下的前端开发具备更强的**可维护性与可把控性**，更适配大型复杂项目。

③Vue 深度依赖自身的响应式更新体系，在部分场景下对原生 JS 的兼容性适配成本略高；而 React 本身无框架黑魔法、贴合原生 JS 的设计特性，天然具备更优异的**跨端兼容能力**，与原生 JS 生态的适配性、兼容性表现更出色。

④两者无优劣之分，只是设计哲学的取舍：Vue 用**「框架的封装便捷」**换开发效率，React 用**「手动的可控约束」**换工程化能力，均是当前前端领域的最优解之一。

options api 代码书写逻辑简单，适合小型项目，但是由于逻辑分散，大型项目不好维护

composition api优势：

### 三、响应式数据声明（ref、reactive）

**ref**：声明简单数据类型，在script标签中使用需要.value进行操作，ref将声明的简单数据类型包装成了一个特殊的对象

**reactive**:声明复杂数据类型，无需.value操作

**computed**：计算属性，基于已有响应式值进行计算，默认有缓存能力，只读属性

**watch**：侦听变化，传入数据源(count),回调函数（newCount,oldCount）=>{操作}

**immediate: true** 默认关闭， 初始化时执行一次

**deep：true**，默认关闭，开启之后支持引用数据类型的属性

问题：如果代码中多个ref和多个watch，如何让代码组织更清晰呢？

```
可以使用reactive合并ref声明的简单数据类型，watch的话可以使用数组监听多个数据或者监听响应式对象里面的属性，需要开启deep：true属性进行深层监听
```



在庞大项目组织结构里面，composition api会给项目结构设计带来怎么样的帮助？

```
Composition API 为庞大 Vue 项目的结构设计带来本质提升：它将代码组织逻辑从「按语法选项拆分」升级为「按业务逻辑聚合」，彻底解决了逻辑碎片化、复用困难、组件臃肿的核心痛点；通过自定义 Hooks 实现无耦合的逻辑复用，形成标准化的项目架构；完美契合 TS 实现强类型约束，保障项目健壮性；同时统一了编码规范，降低团队协作成本。最终让庞大的 Vue 项目具备极高的可维护性、可扩展性、复用性，是大型 Vue 项目的最优架构选择
```

### **五、响应式底层原理--Proxy&defineProperty**---都只是信息源告诉我们`数据变化`，但是通知谁去更新需要`依赖追踪（视图变化）`

①**为什么需要响应式？**

在传统原生 JS 开发模式中，修改数据后需要**手动频繁操作 DOM** 来完成页面更新；如果一处数据变更需要同步更新页面多个位置，就会出现大量重复的 DOM 操作代码，不仅冗余繁琐，还容易出现数据与视图不一致的问题。而 Vue 的响应式系统，能**自动监听数据的变化，并在数据更新时，统一完成视图的联动刷新**，无需开发者手动操作 DOM。这一机制极大降低了开发的心智负担，让开发者可以脱离 DOM 操作的细节，将核心精力聚焦于业务逻辑的开发与维护上。

**②如何做到响应式**

**proxy（拦截行为）**：将每个对象new成proxy对象，然后通过handler对象里面的get（读取代理对象时触发）和set方法（在我们写入或修改代理对象的时候触发）

### 完整proxy实现响应式（proxy,get,track,set,trigger）

#### ✔️ 1. 为什么用两个哈希表？

因为响应式的依赖关系是 **「对象 → 属性 → 副作用函数」** 的三层结构，单层哈希表存不下这种精准的映射关系，必须用「双层嵌套」。

```
targetMap = {
  state: Map = {  //depsMap
    'count': Set { 渲染count的函数 },
    'name': Set { 渲染name的函数 }
  },
  person:Map = {
  	'age': Set{ 渲染age的函数},
  	'gender': Set{ 渲染gender的函数}
  }
}
```

#### ✔️ 2. 两个哈希表的分工（固定搭配）

- 第一层 `WeakMap(targetMap)`：**区分不同的对象** → key = 目标对象 (t)，value = 第二层 Map；
- 第二层 `Map(depsMap)`：**区分同一个对象的不同属性** → key = 属性名 (k)，value = 存副作用函数的 Set；

#### ✔️ 3. track 和 trigger 的核心逻辑

- **track 收集依赖**：顺着「对象→属性」的路径，把副作用函数存到对应的 Set 里；
- **trigger 触发更新**：顺着「对象→属性」的路径，把 Set 里的副作用函数取出来执行；

#### ✔️ 4. 你的代码里的对应关系

你 Proxy 里的 `t` 就是 WeakMap 的 key，`k` 就是第二层 Map 的 key，`activeEffect` 就是 Set 里的内容，**所有参数都能对应上**

```
<p id="count">0</p>
<button id="icon">+1</button>
<script>
// 1. 全局变量：存「当前正在执行的副作用函数」
let activeEffect = null;
// 2. 依赖集合：key是数据对象，value是「属性名: 副作用函数数组」
const targetMap = new WeakMap();


// --------------- track：收集依赖 ---------------
function track(target, key) {
  // 只有当前有activeEffect（有副作用函数在执行），才收集
  if (!activeEffect) return;
  
  ----------------第一层：targetMap（weakMap）键：对象，值：属性集合---------------
  // 给当前数据对象（target）建一个「属性-副作用」的映射
  let depsMap = targetMap.get(target)
  if(!depsMap){
  	depsMap = new Map()
  	targetMap.set(target,depsMap)
  }
  
  ----------------第二层：depsMap（Map）键：属性，值：使用该属性的函数的set集合--------------------
  // 给当前属性（key）建一个「副作用函数数组」
 let deps = depsMap.get(key)
 if(!deps){
 	deps = new Set()
 	depsMap.set(key,deps)
 }
  
  ----------------第三层：值：使用该属性的函数的set集合--------------
  // 把当前副作用函数（activeEffect）加入依赖集合
  deps.add(activeEffect);
}


// --------------- trigger：触发更新 ---------------
function trigger(target, key) {
  // 从依赖集合里，找到当前数据+属性对应的副作用函数
  const depsMap = targetMap.get(target)
  if(!depsMap)return;
  const deps = depsMap.get(key)
  if(!deps)return;
  
  // 执行所有依赖的副作用函数（比如重新渲染页面）
  deps.forEach(effect => effect())
}


// --------------- effect：包装副作用函数 ---------------
function effect(fn) {
  // 把副作用函数设为「当前活跃的effect」
  activeEffect = fn;
  // 执行副作用函数（执行时会读取数据，触发track收集依赖）
  fn();
  // 执行完后清空，避免后续乱收集
  activeEffect = null;
}


// --------------- 你的Proxy代码，加上track/trigger ---------------
const state = { count: 0 };
const p = new Proxy(state, {
  get(t, k) {
    // 读取数据时，收集依赖（记录“谁用了这个数据”）
    track(t, k); 
    return t[k];
  },
  set(t, k, v) {
    t[k] = v;
    // 修改数据时，触发更新（通知“用了这个数据的人”更新）
    trigger(t, k); 
    return true;
  }
});


// --------------- 用effect包装“渲染页面”的副作用函数 ---------------
effect(() => {
  // 这个函数的作用是“渲染页面”，会读取p.count（触发get→track）
  document.getElementById('count').innerText = p.count;
});


// --------------- 点击事件 ---------------
document.getElementById("icon").onclick = function() {
  p.count++; // 修改数据→触发set→trigger→执行effect里的渲染函数→页面更新
};
</script>
```

### vue响应式更新vs React状态更新

①vue和React都默认有批处理机制 【攒一堆修改，一次性渲染】

vue是通过将所有任务放在同一个微任务队列当中，当多次操作数据结束之后，一次性清空微任务队列完成数据更新

强制单独处理：nextTick()  在渲染完再执行后续函数

```
const handleClick = async () => {
  // 1. 连续修改3次状态，Vue把更新任务放进微任务队列
  state.count++;
  state.name = "张三";
  state.age = 20;

  // 2. 调用nextTick，把回调放进同一个微任务队列
  await nextTick();

  // 3. 这个回调会在「所有状态更新完成、视图渲染后」执行
  console.log("此时页面已经是最新的了");
};
```



 react是通过render+commit事件批量处理的，可以使用fluhsync分开处理每一次渲染

```
render 阶段：收集所有状态修改，批量计算「哪些组件需要更新」（生成新的虚拟 DOM）；
commit 阶段：一次性把所有需要更新的组件，渲染到真实 DOM 上。
```

强制单独处理：flushSync  不批量处理修改任务，一次一次更新渲染

```
import { flushSync } from 'react-dom';

const handleClick = () => {
  // 这两次修改会被批处理，合并渲染1次
  setCount(c => c+1);
  setName("张三");

  // 这个修改会「立即单独渲染」，不参与批处理
  flushSync(() => {
    setAge(20);
  });
};
```

### vue的副作用清理逻辑

定时器没关、订阅没关、请求没收,下一次副作用执行之后上一次副作用的定时器还在启动，占用后台资源

副作用函数：watch、watchEffect，它们监听的数据变化之后会执行一次

什么时候清理（谁创建、谁清理）？  1、副作用函数准备下一次重新执行之前；2、组件被卸载

场景1、定时器的清理：onInvalidate

```
<script setup>
import { watchEffect } from 'vue';

// 用onInvalidate清理定时器
watchEffect((onInvalidate) => { // 注意：onInvalidate是watchEffect回调的参数
  // 1. 开启定时器
  const timer = setInterval(() => {
    console.log('定时器运行中...');
  }, 1000);

  // 2. 注册清理函数：当副作用被清理时，执行这个函数
  onInvalidate(() => {
    clearInterval(timer); // 关定时器
    console.log('定时器已被清理');
  });
});
</script>
```

场景2、竞态请求：同时发送多个请求，如果没有关闭，那么一个较早的、响应较慢的请求将会覆盖掉最新的请求，导致数据错误

可以在watch的onInvalidate当中执行abortController.abort()停止请求：在下一次请求之前关掉上一次请求

```
<template>
  <div>
    <input v-model="keyword" placeholder="输入关键词搜索" />
    <div v-if="loading">加载中...</div>
    <div v-else>{{ result }}</div>
  </div>
</template>

<script setup>
import { ref, watch } from 'vue'
import axios from 'axios'

// 搜索关键词
const keyword = ref('')
// 接口返回结果
const result = ref('')
// 加载状态
const loading = ref(false)

// 定义AbortController实例（用于控制请求）
let abortController = null

// 监听关键词变化，发起请求
watch(
  keyword,
  async (newKeyword) => {
    // 1. 清空旧结果，显示加载中
    result.value = ''
    loading.value = true

    // 2. 终止上一次未完成的请求
    if (abortController) {
      abortController.abort() // 终止旧请求
    }

    // 3. 创建新的AbortController
    abortController = new AbortController()
    // 获取signal（用于关联请求）
    const { signal } = abortController

    try {
      // 4. 发起请求（携带signal）
      const res = await axios.get('https://api.example.com/search', {
        params: { keyword: newKeyword },
        signal // 关键：将signal绑定到请求
      })

      // 5. 请求成功，更新结果
      result.value = res.data
    } catch (err) {
      // 捕获“请求被终止”的错误（正常流程，无需提示）
      if (err.name !== 'CanceledError') {
        console.error('请求失败：', err)
      }
    } finally {
      // 6. 无论成功/失败，关闭加载状态
      loading.value = false
    }
  },
  {
    // 7. 在watch的onInvalidate中，终止请求（兜底）
    onInvalidate() {
      if (abortController) {
        abortController.abort()
      }
    }
  }
)
</script>
```

## 缓存策略

概念、资源存储、特定场景（比如刷新之类的）

### bfcache

浏览器自动开始，为同一域名的前进后退网页资源做一个“全量缓存”，但是可能会出现一些问题，比如安全问题（用户输入框中的密码也缓存了）、数据不同步的问题。。。开发者需要通过监听`pageShow和pageHide`做一些清理敏感数据的措施，以及在用户回退的时候请求新的数据

bfcache 的核心是 **“缓存页面完整状态，加速前进 / 后退切换”**，开发者可以：

1. 利用它提升页面切换体验；
2. 通过 `pageshow/pagehide` 事件处理缓存恢复的逻辑；
3. 避免操作导致 bfcache 失效，最大化发挥其性能优势。



### Memory Cache

1. **Memory Cache 核心价值**：浏览器在内存中临时缓存资源，是最快的缓存方式，优先于磁盘缓存；
2. **管理规则**：浏览器自动缓存、按 LRU 淘汰、标签页关闭即清空；
3. **验证方法**：Chrome Network 面板看 `from memory cache`；
4. **优化技巧**：减小资源体积、复用资源、合理设置缓存头，让高频资源留在内存中；
5. **关键认知**：开发者无法手动操作 Memory Cache，只能通过优化资源让浏览器更高效地利用它。

简单记：Memory Cache 是浏览器给前端的 “免费性能红利”，你只需要做好资源优化，就能让它帮你提升页面加载速度。

## 0117

vue2的`Object.defineProperty`对已存在的每个属性使用set/get劫持监听，不能对新增属性监听并且不能无法监听数组长度/索引值修改（没有get/set），所以对数组的方法进行了重写

总结：

Vue2 的 “劫持” 不是 “全量劫持”，而是**针对性拦截**：对对象劫持「已有属性的 get/set」，对数组劫持「7 个可变方法的调用」；

所谓 “无法劫持内部属性”，本质是 `Object.defineProperty` 的天然限制（只能拦截已有属性的读写，无法拦截新增 / 删除 / 数组索引）；

Vue2 响应式的 “补丁方案”（`this.$set`/`this.$delete`、数组用 splice 改索引），都是为了绕开这些 “无法劫持” 的场景，手动触发响应式。



vue3基于proxy监听整个对象，拦截内部方法，不需要动原型，但是比如改动数组长度相当于删除做了处理

### 总结

1. 函数重载的核心是「**多类型签名匹配**」：为同一个函数定义多个「入参 - 返回值」类型规则，TS 按入参类型匹配对应的规则；
2. 结构上分为「重载签名（多个，仅声明）」和「实现签名（一个，实际逻辑）」，实现签名必须兼容所有重载签名；
3. 相比单纯的联合类型，重载能**精准推导返回值类型**，让类型校验更严格、更贴合业务逻辑。

### 冒泡/捕获

捕获是「从上到下」（祖先→目标），冒泡是「从下到上」（目标→祖先），事件执行顺序：先捕获→目标元素→后冒泡；

冒泡是默认行为，核心用于**事件委托**；捕获用于**提前拦截事件**；

大部分 UI 事件（click、input 等）支持冒泡 / 捕获，`focus`/`blur`/`mouseenter` 等不支持；

`e.stopPropagation()` 可阻止事件传播，区分「阻止传播」和「阻止默认行为」（`preventDefault`）。
